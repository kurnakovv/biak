## Summary
Add support for an InspectCode diagnostics baseline feature so users can snapshot the current diagnostics state and keep it synchronized over time.

## Problem
Projects often adopt stricter analysis incrementally. Without a baseline mechanism, existing diagnostics make it difficult to distinguish newly introduced issues from already-known ones. This creates noise in CI and slows down adoption of formatting and analyzer enforcement.

## Proposed solution
Introduce an `inspectcode-baseline` command family for InspectCode integration with two primary commands:

- `init` - generate an initial baseline from current InspectCode output
- `sync` - refresh an existing baseline to match the current accepted diagnostics set

This command family is conceptually similar to `warnings-baseline`, but with a different focus:
- `warnings-baseline` is focused on .NET compiler/analyzer warnings
- `inspectcode-baseline` is focused on InspectCode rules and diagnostics

Baseline behavior must write existing rules using a fixed severity level defined by user configuration (default: `suggestion`). The same severity mapping must be applied consistently during both `init` and `sync`.

## Configuration
Add a dedicated `inspectCodeBaseline` section in biak config.

Example:

```json
{
  "inspectCodeBaseline": {
    "target": "./biak.slnx",
    "path": ".biak/.editorconfig-InspectCodeBaseline",
    "snapshotSeverity": "suggestion",
    "ruleIdOverrides": {
      "CanReplaceCastWithLambdaReturnType": "resharper_can_replace_cast_with_lambda_return_type_highlighting"
    },
    "additionalArgs": [
      "--settings=./team.DotSettings",
      "--no-build"
    ]
  }
}
```

Fields:
- `target` - explicit target passed to InspectCode (`.slnx` / `.sln` / `.csproj`); if not set, use default auto-discovery
- `path` - target baseline file path in config (used when CLI flag is not provided)
- `snapshotSeverity` - fixed severity applied to rules written during `init`/`sync` (default: `suggestion`)
- `ruleIdOverrides` - custom mapping dictionary (`ruleId -> inspectcodeRuleKey`) to add missing rules or override existing mappings
- `additionalArgs` - raw arguments passed to InspectCode for advanced scenarios

## Scope
- Define baseline file format for storing accepted InspectCode diagnostics (with severity)
- Add command-line support for baseline initialization (`init`)
- Add command-line support for baseline synchronization (`sync`)
- Compare current InspectCode results against stored baseline entries
- Surface only new or unresolved diagnostics outside the accepted baseline where appropriate
- Apply configurable fixed severity when writing existing baseline rules (default: `suggestion`)
- Support baseline target path in both CLI and config with precedence: `--path` flag -> `inspectCodeBaseline.path` -> marker-based discovery
- Pass user-defined `inspectCodeBaseline.additionalArgs` through to InspectCode baseline operations
- Ensure behavior works with existing biak concepts such as modular configuration, imports, variables, and enable/disable modes

## Proposed CLI

```bash
biak inspectcode-baseline init
biak inspectcode-baseline sync [--path <path>]
```

## Init/Sync behavior
### `init`
- Resolve InspectCode target from `inspectCodeBaseline.target` or default auto-discovery (`*.slnx` -> `*.sln` -> `*.csproj`)
- Run InspectCode (with configured `inspectCodeBaseline.additionalArgs`)
- Parse diagnostics and map rules to `.editorconfig` keys
- Apply `inspectCodeBaseline.ruleIdOverrides` over built-in mappings
- Produce baseline entries with fixed severities according to `inspectCodeBaseline.snapshotSeverity`
- Print baseline blocks in editorconfig-style format, marked with `# ^biak^ inspectcode-baseline`

Conceptual output shape:

```text
[{FilePath1,FilePath2}]
ruleId = severity # ^biak^ inspectcode-baseline
...
```

### `sync`
- Require root `.editorconfig` in current directory for runtime analysis
- If `.biak` exists, require synchronized biak status (`enabled` or `disabled`)
- Resolve target baseline path with precedence: CLI `--path` flag -> `inspectCodeBaseline.path` from config -> discovery by marker in `.biak/**/.editorconfig*` -> root `.editorconfig`
- Validate path safety (inside current project, file name starts with `.editorconfig`, not under `.editorconfig` directory)
- If baseline marker is not found anywhere, return a user-facing message asking to run `biak inspectcode-baseline init`
- Run InspectCode and compare current diagnostics to baseline entries
- Remove resolved filters and stale file entries
- Normalize remaining baseline entries to `inspectCodeBaseline.snapshotSeverity`

## Acceptance criteria
- A user can create a baseline from existing InspectCode diagnostics using `init`
- A user can update the baseline using `sync` after intentional diagnostics/rule changes
- Baseline matching is deterministic and stable across runs for the same input
- Newly introduced diagnostics can be distinguished from diagnostics already present in baseline
- Existing rules are written with fixed configured severity from `inspectCodeBaseline.snapshotSeverity` (default: `suggestion`)
- `inspectCodeBaseline.ruleIdOverrides` supports adding custom mappings and overriding built-in mappings
- If a mapping exists in both built-in mappings and `ruleIdOverrides`, the config override is used
- Path resolution precedence is deterministic and enforced as: CLI `--path` flag -> `inspectCodeBaseline.path` -> marker-based discovery
- `inspectCodeBaseline.additionalArgs` are forwarded unchanged and in the specified order
- Baseline entries use editorconfig-style format with marker `# ^biak^ inspectcode-baseline`
- If marker is not found anywhere, `sync` asks user to run `biak inspectcode-baseline init`
- The baseline workflow is documented with examples, including config usage
